## Phase 1: 전문가별 분석

### Design Director
- Toast의 정체성은 레이아웃 구조가 아닌 **UX 행동**: 일시적, 자동소멸(2-5초), 비차단 피드백
- UX Notification Spectrum: Toast(일시) < Banner(지속) < Alert Dialog(필수 응답)
- 사용자가 추려놓은 노드 판별:
  - Bar/Status/ChallengeWIP → Banner (조건 충족 전까지 상시 노출)
  - 선택지 리스트 (가족/연인/친구) → List/Selection UI (알림 패턴 아님)
  - 친구 리스트 개별 노드 → List Item (알림 패턴 아님)
- 새 컴포넌트 추가 기준: 현재 온보딩에 Toast 사용처 0건, 기존 피드백은 Alert Dialog/Bottom Sheet/풀스크린으로 커버

### Engineering Lead
- Sonner의 기술적 본질: 전역 이벤트 시스템 + 자동 라이프사이클 + 스택 관리 + Promise 통합
- Figma 컴포넌트보다 **코드 구현체의 행동**이 핵심
- shadcn/ui도 기존 Toast를 deprecated → Sonner로 대체
- 앱 전체에선 필요하지만, 온보딩에 사용처 없는 현재 추가는 시기상조
