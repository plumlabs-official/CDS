# Lounge Updates Reply Preview Component Decision

- **Date**: 2026-05-08
- **Mode**: Team Mode - Feature Review / Component Architecture Decision
- **Roles**: Product Leader, Design Director, Engineering Lead
- **Topic**: Whether the new reply-preview UI for lounge update items should be added as a new Lounge Card variant or as a separate composed component.

---

## Evidence

- Existing node `25945:38066` is a remote CDS `INSTANCE` of component set `Lounge Card`.
- Existing `Lounge Card` component set has `State=1Line/2Line`, 2 variants, and properties/slots including `Title`, description/date/timestamp booleans, `Show Thumbnail`, `Show Info Slot`, `Challenge Sub List`, `Thumbnail Slot`, and `Info Slot`.
- Proposed node `28047:38198` is a `FRAME`, not a CDS instance. It keeps the top card and challenge sub list, then adds `Reaction Bar` and `Reply Preview`.
- Proposed frame uses mostly CDS instances, but the reply count icon is `Phosphor Icons / chat-centered-dots`, which conflicts with the CDS Lucide icon convention.

## Phase 1: Expert Analyses

### Product Leader

Using a user-problem lens, the new UI solves a narrower problem than the base `Lounge Card`: helping creators notice and preview new comments on their own updates. That is a feed/notification behavior, not just a card display state. If this is placed inside the base `Lounge Card`, the component starts absorbing workflow-specific behavior for all lounge cards.

Recommendation: keep the base card reusable and create a product-level update item composition for this feed use case.

### Design Director

Using design system thinking, the proposed UI introduces two new functional zones: engagement actions and reply preview. That changes the organism structure from "card with optional slots" to "feed item with engagement and latest activity." Adding this as `State=NewReplies` to the existing `State=1Line/2Line` axis would mix content density with notification state.

Recommendation: create a new composed component that nests `Lounge Card`, then adds `.Reaction Bar` and `.Reply Preview`. Replace the Phosphor icon with a Lucide-based reply/message icon.

### Engineering Lead

Using modular design, comment preview data should not become part of the base card API. The existing `Lounge Card` already has a broad property surface. Adding like/reply counts, commenter avatars, commenter names, comment bodies, preview visibility, and row counts would push it toward prop bloat and variant/property ambiguity.

Recommendation: compose instead of extending. In implementation terms, map this to `LoungeCard` plus `ReactionBar` plus `ReplyPreviewList`, with a wrapper such as `LoungeUpdateItem`.

## Phase 2: Synthesis

### Agreements

- Do not add this directly as a new variant on the existing `Lounge Card`.
- Reuse the existing `Lounge Card` internally so the visual language and existing slots are preserved.
- Create a new composed component for the feed/news-item use case.
- Fix the non-CDS Phosphor icon before componentization.

### Conflict

- Design system reuse would normally favor extending the existing component because most of the top structure is shared.
- Engineering and product concerns outweigh that here because the new behavior adds feed-specific data and interaction zones.

### Recommendation

Create a new composed component, tentatively named `Lounge Update Item` or `Lounge Updates Item`, placed in the relevant Lounge Cards context group. It should nest the existing `Lounge Card` and add controlled lower sections for reactions and reply preview.

Use a boolean such as `Show Reply Preview` or a slot such as `Reply Preview Slot` inside the new wrapper, rather than adding a `New Replies` variant to the base `Lounge Card`.
