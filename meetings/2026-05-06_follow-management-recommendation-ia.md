## Phase 1: 전문가별 분석

### Growth Expert

Growth Loop 관점에서 현재 우선순위는 피드/라운지에서 볼 사람을 찾고 팔로우/초대하게 만드는 것이다. `팔로우 관리`는 성장 액션이 아니라 관계 정리와 통제감을 위한 guardrail이다.

결론:

- `관리 -> 추천` 이동은 허용한다.
- `추천 -> 관리` 이동은 기본 UI에서 제거한다.
- 추천 화면의 첫 화면은 팔로우/초대/연락처 연결에 집중해야 한다.

### Product Leader

두 화면은 해결하는 문제가 다르다.

- `친구 찾기`: 볼 사람을 찾지 못해 피드/라운지 가치가 낮은 문제를 해결한다.
- `팔로우 관리`: 관계가 많아진 뒤 피드 품질과 통제감을 관리하는 문제를 해결한다.

결론:

- 비대칭 연결은 문제 없다. 과업 자체가 비대칭이기 때문이다.
- 추천 화면의 canonical name은 `친구 찾기`로 고정한다.
- 관리 화면은 프로필 수치, 설정, row overflow에서 접근 가능해야 한다.

### Design Director

화면의 정보 향을 기준으로 보면 추천 화면은 `누구를 팔로우할지`, `왜 추천됐는지`, `지금 팔로우/초대할 수 있는지`가 중심이어야 한다.

결론:

- 추천 화면에는 팔로우 관리 버튼을 두지 않는다.
- 팔로잉 친구 preview도 두지 않는다.
- 기존 팔로잉 정보는 추천 카드 내부 social proof로만 사용한다.

### Data Scientist

이 결정은 추천 화면 전환율과 관리 기능 발견성 사이의 trade-off다.

Primary metrics:

- 추천 화면 팔로우 완료율
- 초대 완료율
- 첫 팔로우까지 걸리는 시간
- 피드/라운지 첫 유효 상호작용 도달률

Guardrails:

- 팔로우 관리 발견 실패율
- 숨기기/끊기/차단 완료 시간
- 추천 dismiss율
- 추천 화면 이탈률

## Phase 2: 종합 토론

### 합의점

- 추천과 관리는 같은 화면에서 섞지 않는다.
- 현재 성장 맥락에서는 `친구 찾기/추천/초대`가 primary flow다.
- `팔로우 관리`는 secondary utility로 낮춘다.
- `팔로우 관리`에는 `친구 찾기`로 가는 가벼운 출구를 둔다.
- `친구 찾기`에는 `팔로우 관리` 버튼이나 팔로잉 preview를 두지 않는다.

### 충돌점

- Growth는 추천 화면의 관리 진입을 강하게 반대했다.
- Product는 관리 기능을 너무 숨기면 신뢰/통제감 문제가 생길 수 있다고 봤다.
- Design은 비대칭 내비게이션의 일관성 우려보다 과업 선명도를 우선했다.
- Data는 A/B 테스트 가능성을 열어두되 기본값은 A+D로 보는 데 동의했다.

### 트레이드오프

| 선택지 | 장점 | 단점 | 결정 |
|---|---|---|---|
| 관리 -> 추천 버튼 | 성장 액션으로 복귀 가능 | 비대칭 구조 | 적용 |
| 추천 -> 관리 버튼 | 이동 편의성 | 추천 전환율/주의 분산 리스크 | 제외 |
| 추천 화면 팔로잉 preview | 관계 그래프 느낌 | 공간 낭비, 목적 혼합 | 제외 |
| 추천에 관리 노출 없음 | 목적 선명 | 관리 이동 우회 | 적용 |

### 추천

`Option A + Option D`를 채택한다.

구체적으로:

1. `팔로우 관리` 우측 상단에는 `UserPlus` / `사람 추가` 아이콘을 둔다.
2. 해당 아이콘은 `친구 찾기`로 이동한다.
3. `친구 찾기` 우측 상단에는 `팔로우 관리` 버튼을 두지 않는다.
4. `친구 찾기`에는 팔로잉 친구 preview 리스트를 두지 않는다.
5. 이미 팔로우 중인 추천 카드는 `팔로잉` 상태만 보여주고 source label은 숨긴다.
6. `친구 찾기` 구현 전 CDS `Recommendation Profile List`를 별도 composed pattern으로 만든다.
