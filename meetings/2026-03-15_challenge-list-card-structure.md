## Phase 1: 전문가별 분석

### Design Director

Atomic Design + 디자인 시스템 준수 기준 적용.

구조 비교:
- 기존 Challenge Card: Content Header + Button + Badge (TDS 프리미티브 조합), 챌린지 진행 상태 표시
- 챌린지 리스트 카드: Avatar + Image BG + Title Overlay + Stats + Badge, 챌린지 탐색/피드

공유 프리미티브가 Avatar 하나뿐. 레이아웃 철학 상이 (Content Header 기반 vs Image-first 오버레이).

새 컴포넌트 추가 기준:
- 3회 이상 재사용 예상? → Yes (피드에서 N개 반복, 미니 카드 4개)
- 기존 컴포넌트로 대체 가능? → No

추천: 별개 컴포넌트로 분리.

### Engineering Lead

바이브코딩 매핑 + Modular Design 관점.

Modular Design 평가:
- 통합 시: 응집도 낮음 (3가지 레이아웃 한 파일), 결합도 높음, AI 매핑 부정확
- 분리 시: 응집도 높음 (단일 책임), 결합도 낮음, 이름만으로 명확

Start Simple 원칙 적용 → 구조가 다른 것을 합치면 오히려 복잡. 분리가 단순.

추천: 별개 컴포넌트 분리 + 네이밍 명확화.

## Phase 2: 종합 토론

### 합의점
- 별개 컴포넌트로 분리 (두 역할 모두 동의)
- 구조 차이 근본적 (Content Header 기반 vs Image-first 오버레이)
- 바이브코딩 AI 매핑 정확도를 위해 이름으로 역할 구분 필수

### 충돌점
- 네이밍 방식: Challenge 접두사 vs 역할 기반 (FeedCard)

### 트레이드오프

| 옵션 | 장점 | 단점 |
|------|------|------|
| A. Challenge 접두사 유지 | 도메인 명확, 패밀리 인식 | 이름 길어짐 |
| B. 역할 기반 (FeedCard) | 짧고 범용적 | 챌린지 전용인지 불명확 |

### 추천

별개 컴포넌트로 분리 + Challenge 패밀리 네이밍 유지.

최종 구조:
| 컴포넌트 | TDS 분류 | 설명 |
|----------|---------|------|
| Challenge List Card | Composed | 크리에이터 풀사이즈 피드 카드 |
| Challenge Mini Card | Primitive | 164×218 그리드용 소형 카드 |
| Challenge Grid | 레이아웃 패턴 (컴포넌트 X) | 화면 배치 시 직접 구성 |
| Challenge Card (기존) | Composed | 변경 없음 |

Challenge Grid를 컴포넌트로 안 만드는 이유:
- 사용처 1곳 (3회 미만 재사용 기준 미달)
- Mini Card + 섹션 헤더 + 버튼 직접 배치로 충분
