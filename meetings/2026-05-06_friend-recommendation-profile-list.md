# Friend Recommendation Profile List

- **Roles**: Product Leader, Design Director, Growth Expert

## Phase 1: Expert Analyses

### Product Leader

Using the Product Strategy framework, the problem is not generic profile
display; it is recommendation trust. A generic follower-count profile card does
not explain why this person is relevant. A dedicated recommendation list is
justified if it carries reason metadata such as mutual follows and source.

Recommendation: create a friend-recommendation profile list pattern, but keep it
as a composed variant around existing Profile Card anatomy where possible.

### Design Director

Using design system consistency criteria, follower count should not be the
primary metadata in recommendation contexts. It makes the row feel popularity
driven. The row should show:

- primary: avatar, name, follow action
- secondary: mutual/social proof (`김예인님 외 3명이 팔로우`)
- tertiary/source: recommendation source (`연락처에서 가져옴`, `함께한 챌린지 기반`)

### Growth Expert

Using Activation Optimization, recommendations need to reduce decision friction.
Mutual context and source labels are stronger activation cues than raw follower
counts because they answer "why should I follow this person now?"

Guardrail: do not over-explain every row. One concise reason line plus a small
source label is enough.

## Phase 2: Synthesis

### Agreements

- A recommendation-specific profile list is better than reusing the current
  follower-count list as-is.
- Hide follower count by default in recommendation contexts.
- Show mutual follow context and recommendation source instead.
- Keep the underlying pattern CDS-compatible and avoid one-off custom rows.

### Conflicts

- Product wants a new pattern only if it solves trust and recommendation quality.
- Design wants a distinct visual hierarchy so the row does not look like a
  generic profile list.
- Growth wants the row to stay lightweight and scannable.

### Recommendation

Create a `Recommendation Profile List` composed pattern with rows based on CDS
`Profile Card` or `Invite Profile Card` anatomy. It should not expose follower
count by default. The recommended metadata hierarchy is:

1. Name
2. Mutual context: `김예인님 외 3명이 팔로우`
3. Source: `연락처에서 가져옴`, `함께한 챌린지 기반`, `내 관심사 기반`
4. Action: `팔로우`

## Phase 3: QA

PASS for concept direction. Needs implementation validation if applied in Figma:
CDS instance usage, no detached custom rows, and source labels present on
recommendation rows.
