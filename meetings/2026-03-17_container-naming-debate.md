## Phase 1: 포지션 선언

### Design Director — Container 제한적 허용
- M3가 2023년에 의도적으로 Container 채택. "레거시"가 아님
- Container ≠ Content. "레이아웃 구조물" vs "내용물"은 다른 의미
- Figma에서 레이아웃 래퍼가 필요한 순간은 많음 (Auto Layout, 스크롤)

### Engineering Lead — Container 전면 금지 유지
- M3는 Web Components 생태계, TDS는 shadcn/Tailwind. 다른 생태계
- shadcn/Radix 59개 컴포넌트 Container 사용 0건
- Anima/bolt.new: 레이어명→className 1:1 매핑. `container` → Tailwind max-width 적용 = 실제 버그
- v0: "AI will use these names to generate component and prop names" — Container 오역 리스크

### AI Ops Expert — 금지 + 대안 정의
- 레이아웃 래퍼 개념은 유효 (DD 동의)
- Container라는 "단어"에 리스크 (EL 동의)
- 대안: Content(기본), Area(순수 레이아웃), Group(동종 묶음)

## Phase 2: 공격적 반론

- DD→EL: "Anima의 멍청한 1:1 매핑이 문제지, Container 단어의 문제가 아니다"
- EL→DD: "바이브코딩에서 레이어명=컴포넌트명. 인간의 도메인 분리는 AI에 없다"
- AI Ops→양쪽: "래퍼가 필요하면 다른 이름으로 부르면 됨. Container 논쟁 자체가 불필요한 래퍼에서 발생"
- DD→AI Ops: "레이아웃 래퍼는 필수 요소. Content로 대부분 커버되고, Area가 나머지 커버"
